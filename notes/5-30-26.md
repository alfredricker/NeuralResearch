## Synapse allocation: pre-allocated slots, sorted at allocation

**Decision: synapses are pre-allocated as fixed slots in phase 1, not created by connections.**

A dendrite's synapse count is fixed by the neuron type (`synapses_per_dendrite`). So in the
allocation phase — alongside somas and dendrites — every dendrite gets its full pool of
*empty synaptic slots*:
- `synapse_xs` sampled from the config sampler **and sorted ascending within each dendrite**
- `synapse_weights` initialized (small uniform)
- `synapse_alphas` = 0, `synapse_last_events` = 0
- `synapse_offsets[d] = d * synapses_per_dendrite` — **analytic**, no prefix-sum pass

Connections (phase 2) then **bind a presynaptic source to an existing empty slot** — they
create nothing. They consume slots from a per-target `SlotCursor` and emit
`(source_neuron, target_synapse)` pairs, which are grouped by source into the axon CSR.

### Why sort at allocation (the whole point)
- The gamma computation in `update_dendrite_activity` *requires* `synapse_xs` ascending
  within each dendrite. Sorting once at allocation — before any wiring — makes that
  invariant **structurally impossible to violate**: phase 2 only assigns sources, it never
  reorders.
- Fixed stride ⇒ offsets are analytic, addressing is predictable, memory access coalesces on
  GPU. This matches the existing offset-array design (`stride = synapses_per_dendrite`).

The alternative (synapses materialized in connection order) would force a post-hoc sort that
co-permutes weights/xs/alphas/last_events together, plus a counting pass to size the arrays —
fragile and slower. Rejected.

## Structural plasticity: can we delete / migrate synapses without a nightmare?

**Short answer: not a nightmare — *if* structural changes never rewrite `synapse_offsets`
in-place at runtime.** The fragility everyone fears comes from one specific mistake; avoid it
and the rest is a clean, well-known pattern (tombstone + batch compaction).

### Why naive in-place deletion *is* a nightmare
Two things make "just remove the synapse and shift everything down" terrible:
1. **`synapse_offsets` is fixed-stride.** Removing an element breaks the uniform stride, so
   every downstream offset would have to be rewritten — O(total synapses) per deletion.
2. **`axon_targets` reference absolute synapse indices.** Deleting/moving a synapse shifts the
   absolute index of every synapse after it, silently invalidating every incoming axon pointer.
   This is the real trap: the damage is non-local and easy to miss.

So: **never delete by shifting at runtime.**

### The simple, non-fragile model
Separate *liveness* from *layout*. Keep the fixed-stride layout immutable during a trial;
represent structural change as cheap local edits, and reconcile globally in a batch pass.

**1. Tombstoning (runtime, O(1)).** Don't remove — mark dead (sentinel weight, a dead flag, or
   `x = 255`). The gamma loop skips dead slots. Layout, offsets, and axon_targets are untouched.
   "Delete a synapse" = flip a flag. No array touched structurally.

**2. Migration within the fixed block (runtime, O(stride)).** A dendrite is a small fixed block
   (~16 slots). To move/insert a synapse: write it into a dead slot in the *target dendrite's
   block* and re-sort that block (≤16 elements — trivial). Keeping live synapses packed at the
   front of each block with a per-dendrite `live_count` lets the gamma loop run over
   `[offset, offset + live_count)` instead of the full stride. All edits stay inside one
   fixed-size block; **offsets never change.**
   - Caveat: a within-block move changes a synapse's absolute index, so any `axon_target`
     pointing at it must be updated. Two clean ways out: (a) make `axon_targets` address a
     *logical* `(dendrite, slot)` rather than a raw flat index, or (b) defer the fixup to
     compaction (below) and accept a brief stale-pointer window only for migrated synapses.

**3. Compaction (offline, between epochs, single linear pass).** When tombstones accumulate /
   fragmentation grows, rebuild rather than patch:
   - walk dendrites, copy live synapses into a fresh array, recompute `synapse_offsets`
   - build an `old_index -> new_index` remap table in the same pass
   - apply the remap to `axon_targets` (one linear scan)

   This is the GC model. It is *not* fragile precisely because it rebuilds from scratch with an
   explicit remap, instead of incrementally mutating a structure that other arrays point into.

### Recommendation
Commit to: **fixed stride + capacity, liveness via flag/`live_count`, never rewrite offsets at
runtime, batch-compact offline with an index remap applied to `axon_targets`.** This preserves
the GPU-friendly analytic layout, makes runtime structural plasticity O(1)–O(stride) and local,
and confines the one genuinely global concern (axon pointer remap) to a deliberate, testable
batch pass.

If structural plasticity ever needs *more* synapses on a dendrite than the config stride
allows, that's a separate decision: over-provision each block with headroom slots at allocation
(stride + slack) so growth fills headroom without reallocation, and only compaction reclaims it.
